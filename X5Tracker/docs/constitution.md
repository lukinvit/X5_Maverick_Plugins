> Адаптировано из X5_BM mothership@c87e9ee6 (2026-06-13). Ядро портировано, слой локализован под MCP-плагины/XTracker.

**Статус:** canonical · верховный закон программы «X5 Maverick Plugins». Единый свод неизменных принципов, из которых выводятся все архитектурные, продуктовые и процессные решения по MCP-плагинам для платформы Jarvis / X5 Maverick.
**Версия:** 1.0.0 (2026-06-17)
**Применяется к:** оркестратору (Claude в чате с владельцем), всем агентам команды (`xt-builder`, `xt-protocol-qa`, `xt-scribe`), а также людям-контрибьюторам.
**Модель документа:** apex + ссылки. Устав владеет **принципами** и **индексом доменных канонов**; детали домена живут в своём каноне (Часть II). При конфликте действует precedence из Части III.

---

## Преамбула — две супер-ценности

XTracker-плагин — это мост между чатом Jarvis и внутренним трекером X5 (issue/project-трекер класса Jira/YouTrack, `pulsar.x5.ru`). Плагин держит в руках **секреты доступа** к рабочим данным организации и **мутирует рабочие сущности** (тикеты, статусы, комментарии). Из этого выводится всё остальное:

> **1. Данные и секреты sacred.** Данные трекера пользователя и секреты инстанса нельзя терять, искажать, **дублировать** или **утекать**. Утечка токена/конфига или порча/дубль тикета — катастрофа, а не баг.
>
> **2. Только production-ready.** Никаких заглушек/MVP/«временно» как финала. Попало в `main` → работает end-to-end под реальным инстансом XTracker, с протокол-тестами и сверкой контракта API.

Каждый из 16 инвариантов Части I — операционализация одной или обеих супер-ценностей. **Нарушение любого инварианта = блокер** (мержа, деплоя, закрытия тикета), пока не устранено или не проведена явная поправка устава (Часть III).

## Как пользоваться

- **Перед любым архитектурным/продуктовым решением** — пройди список Части I. Предложение, нарушающее хотя бы один инвариант, либо отвергается, либо требует явной поправки устава (не молчаливого обхода).
- **Перед мержем/деплоем/закрытием тикета** — проверь, что работа не нарушает ни одного инварианта; для процессных (P-13…P-15) это означает реальное evidence, а не заявление.
- **Spec-driven**: stage-2 audit / ревьюер сверяют новую спеку с Частью I через **Constitution Check** ([constitution-check.md](constitution-check.md)) — связывающий гейт; любой `MUST-FLAG` блокирует.
- **Агенты** сверяются с уставом перед действием; их операционные правила (`bug-handling-process.md`, `AGENTS.md`, профили агентов) — подчинённые конкретизации, не заменяют устав.

## Часть I — 16 инвариантов

Формат: **MUST/SHOULD-формулировка** · _Зачем_ · _Как проверить_ · _Источник_.

### 🔒 Данные и секреты

#### P-1 — Secrets Sovereignty
**MUST.** Секреты XTracker (token / password / api-key) приходят **только** через `__jarvis.config`. Никогда — из env, кода, логов, `SKILL.md`, manifest, `structured_content` или текста, уходящего в LLM. В сообщениях об ошибках секрет маскируется.
- _Зачем:_ супер-ценность №1; секрет в логе/issue/контексте LLM — утечка доступа ко всему трекеру организации.
- _Как проверить:_ grep на хардкод-секреты пуст; `xt-protocol-qa` подтверждает отсутствие значения секрета в любом `content`/`_meta`/`runtime.log`.
- _Источник:_ I-1/I-7 mothership → spec §7.3.

#### P-2 — Data Safety
**MUST.** Необратимые/деструктивные операции (delete issue, delete comment, mass-transition) — **не дефолт**; требуют явного флага/подтверждения. Плагин не теряет и не дублирует данные пользователя.
- _Зачем:_ супер-ценность №1; «тихая» массовая мутация рабочих данных невосстановима.
- _Как проверить:_ деструктивные tools требуют explicit-параметр (`confirm`/`force`); нет батч-мутаций без явного намерения.
- _Источник:_ I-2 mothership.

#### P-3 — Idempotent-Skip
**MUST.** Если операция уже выполнена (дубль, повторный `operation_id`, idempotency-конфликт) — возвращай **успех** (`isError:false`) с маркером `⏭️ ПРОПУСК` + явным «НЕ ПОВТОРЯЙ», а **не** ошибку.
- _Зачем:_ слабые LLM трактуют `isError:true` как «retry с другими args» → бесконечный loop. `isError:false` со skip-маркером = модель идёт дальше.
- _Как проверить:_ тест: повторный вызов с теми же args → `isError:false`, состояние не дублируется.
- _Источник:_ spec §21.

### 🏛 Контракт платформы

#### P-4 — Package Contract
**MUST.** В корне zip — валидный `jarvis-plugin.json` (slug `^[a-z][a-z0-9_-]{2,50}$`, `command` без shell, все `*_path` относительные без `..`). В пакете нет `.git/__pycache__/node_modules/.env`.
- _Зачем:_ установщик платформы отвергает или небезопасно распаковывает невалидный пакет; лишние файлы тянут мусор/секреты.
- _Как проверить:_ manifest валидируется против контракта spec §4; `unzip -l` не содержит запрещённого.
- _Источник:_ spec §3–4.

#### P-5 — Protocol Fidelity
**MUST.** Корректный stdio JSON-RPC (initialize / tools/list / tools/call). Процесс **не падает** (один процесс на пакет — краш = недоступность всех tools). Внешние вызовы обёрнуты в try/except с actionable-ошибкой.
- _Зачем:_ упавший процесс роняет весь плагин; невалидный ответ ломает нормализацию в `ToolResult`.
- _Как проверить:_ `xt-protocol-qa` гоняет реальный JSON-RPC: tools/list возвращает валидный result; tools/call на битых args не роняет процесс.
- _Источник:_ spec §6, §18.5.

#### P-6 — Frozen Identifiers
**MUST.** `slug` неизменен между версиями. `remote_name` инструментов не менять (ломает HITL-настройки и сохранённые в чатах actions). Переименование — только новый tool + пометка старого `deprecated`.
- _Зачем:_ смена slug ломает ACL-assignments и tool-namespacing; смена tool-name ломает кнопки в истории чатов.
- _Как проверить:_ diff manifest/tools между версиями — только добавления; нет переименований без deprecated-моста.
- _Источник:_ I-6 mothership → spec §18.11.

### 🛡 Безопасность и изоляция

#### P-7 — Schema Rigor
**MUST.** Каждый tool: `type:object`, строгие `properties` с `description`, где уместно — `enum`/`default`/`required`, `additionalProperties:false`. Неизвестные поля (включая `__jarvis`) извлекаются осознанно или игнорируются, но не ломают обработчик.
- _Зачем:_ нестрогая схема = LLM передаёт мусор, плагин падает или делает не то; promptability держится на описаниях полей.
- _Как проверить:_ `xt-protocol-qa` валидирует каждую запись tools/list; adversarial-вызов с лишним полем не ломает tool.
- _Источник:_ spec §7.2.

#### P-8 — Identity & RBAC Honesty
**MUST.** Плагин действует под сконфигурированной XTracker-identity (сервис-аккаунт / api-key). Не обходит RBAC XTracker. Ошибка прав (403) показывается честно — не маскируется под пустой результат и не «обогащается» данными за пределами прав инстанса.
- _Зачем:_ обход RBAC трекера = утечка чужих рабочих данных; «тихий» 403 вводит пользователя и LLM в заблуждение.
- _Как проверить:_ 403 от XTracker → tool возвращает понятную ошибку прав, не fabricated-данные; нет запросов под чужой identity.
- _Источник:_ I-5/I-7 mothership.

### ⚙️ LLM-эргономика и эксплуатация

#### P-9 — Discovery Quality
**MUST.** `SKILL.md`: YAML `name` + `description` (что делает + когда использовать + триггерные фразы + «НЕ использовать для»; ≤1024 символа, без `<` `>`, на русском). Тело — рабочие процессы прозой/таблицами, **без литеральных вызовов tools** (`tool({...})` / `{"name":...}`). Тело ≤5000 слов.
- _Зачем:_ по `description` LLM решает грузить ли плагин (Progressive Disclosure). Литеральные примеры вызова слабые LLM копируют как текст вместо реального tool_call (инцидент `data_visualization` 2026-05).
- _Как проверить:_ `description` укладывается в лимит, содержит триггеры и негативные триггеры; в теле нет ```json-блоков «как вызвать».
- _Источник:_ spec §5, §18.1–18.2.

#### P-10 — Batch/Loop-Safe
**MUST.** Имена tools — `slug__verb_noun`. Data-провайдеры используют защищённые глаголы (`search_/get_/list_/read_/fetch_`). Action-tools возвращают compact-результат `(N) [ref=X]`. ERROR-результаты содержат keyword (`Error`/`Ошибка`) или `{"type":"ERROR"}`. Виртуальные `__name__` не используются (зарезервированы harness'ем).
- _Зачем:_ harness сжимает/блокирует action-глаголы под batch-нагрузкой; защищённые verb'ы и compact-формат сохраняют смысл в плейсхолдере.
- _Как проверить:_ data-getters названы защищённым verb'ом; action-tools отдают `(N)`-формат; ERROR-формат соблюдён.
- _Источник:_ spec §20.

#### P-11 — Generative UI Discipline
**MUST.** Каждый `structured_content` сопровождается `text`-fallback и **никогда** не попадает в LLM-контекст. Actions имеют уникальный `id` + полное имя `slug__remote_name`. Файлы >5–10KB возвращаются через `upload_file` SDK (artifact_id), внутри content-item `type:"json"`.
- _Зачем:_ structured_content — чисто UI-слой; без text-fallback LLM и клиенты без Generative UI слепнут; крупный base64 раздувает MCP-IPC и context-window.
- _Как проверить:_ у каждого structured_content есть text; actions с полным tool-именем; крупные файлы идут artifact_id, не base64.
- _Источник:_ spec §17, §5.3.

#### P-12 — Resilience
**MUST.** Внешние вызовы имеют таймауты/ретраи. При недоступном/медленном XTracker — понятная ошибка, а не краш. Уважать лимиты платформы (60s на tool-call, 100MB stdout).
- _Зачем:_ заседание/работа пользователя не должны падать из-за временной недоступности трекера; превышение лимитов убивает процесс.
- _Как проверить:_ отключение/таймаут XTracker в тесте → tool отдаёт actionable-ошибку, процесс жив; ответы укладываются в лимиты.
- _Источник:_ I-8 mothership → spec §12.4.

### 🔬 Процесс

#### P-13 — Production-Ready by Default
**MUST.** Каждая поставка production-grade с первого коммита: ноль `stub`/`TODO`/`FIXME`/placeholder как финал; работает end-to-end под реальным инстансом против живого API; покрыта протокол-тестами; задокументирована. Временный workaround — только явно помеченным `# TEMP: <причина>, см. #<issue>`; молчаливая заплатка — нарушение.
- _Зачем:_ супер-ценность №2; «временное» гниёт; полу-рабочий tool в проде хуже отсутствующего.
- _Как проверить:_ нет заглушек в shipped-диффе (или помечены `# TEMP` + issue); протокол-тесты зелёные; tool прогнан сквозным JSON-RPC-раунд-трипом.
- _Источник:_ I-11 mothership.

#### P-14 — Root-Cause + Scope-the-Class
**MUST.** На **любой** баг **любой** критичности — `superpowers:systematic-debugging` прежде фикса: воспроизвести → изолировать → доказать root cause → потом фикс. **Scope-the-class:** спроси «повторяется ли в других местах?» и почини КАЖДУЮ ось: **tool / поле schema / роль XTracker / состояние иссью / поверхность вывода** + поставь guard от регрессии.
- _Зачем:_ односторонний фикс даёт частичное покрытие; симптоматическая правка маскирует более глубокий дефект.
- _Как проверить:_ в тикете есть repro + доказательство root cause; добавлен regression-тест; перебраны все оси класса.
- _Источник:_ I-12 mothership; [bug-handling-process.md](bug-handling-process.md).

#### P-15 — Zero-Trust Verification
**MUST.** Никому не верь на слово — включая агента и его «FIXED», включая собственную память. Любой claim о результате («работает», «tools/list ок», «idempotent») принимается **только** со свежим evidence, добытым проверяющим (реальный JSON-RPC-прогон / тест-вывод / транскрипт), желательно на **другой оси**, чем источник claim'а. Память устаревает — верь живому коду/API.
- _Зачем:_ доверие = дефект процесса; устаревшая память называет несуществующие символы/эндпоинты.
- _Как проверить:_ закрытие тикета сопровождается evidence проверяющего (не разработчика); факт из памяти/доки сверен с живым кодом/API перед использованием.
- _Источник:_ I-13 mothership; [bug-handling-process.md](bug-handling-process.md).

### 🗄 Целостность под объёмом

#### P-16 — Bounded Results
**SHOULD.** Поиск/списки иссью ограничены: явный `limit` (с дефолтом и максимумом) + пагинация по контракту XTracker. Не тянуть тысячи иссью в один tool-результат — это раздувает MCP-IPC и context-window агента и снижает качество планирования LLM. Крупные выгрузки — через `upload_file` (P-11).
- _Зачем:_ неограниченная выдача убивает контекст агента и латентность; bounded-результат — половина инварианта I-15 mothership, применимая к плагину-клиенту.
- _Как проверить:_ у `search_issues` есть `limit` с `maximum`; нет неограниченной выдачи в hot-path.
- _Источник:_ I-15 (bounded queries) mothership → spec §5.3, §20.

## Часть III — Precedence, поправки, enforcement

### Precedence (порядок старшинства при конфликте)

```
Устав (этот файл)  >  платформенный канон (mcp_plugins_dev_spec.md)  >  контракт XTracker API (структура)  >  CLAUDE.md  >  память агентов
```

- Устав задаёт **принципы**; платформенный канон — **детали контракта MCP-плагинов**; контракт API — структуру эндпоинтов XTracker; CLAUDE.md — операционные напоминания; память — контекст эпизодов.
- Память и CLAUDE.md могли устареть (P-15): при расхождении с живым кодом/API — верь живому состоянию.

### Поправки (amendment)

Инвариант **изменяется только явной поправкой**, не молчаливым обходом:

1. PR, изменяющий конкретный инвариант + bump версии устава.
2. Запись в «Журнал поправок» ниже (дата, что изменено, почему, ссылка на PR/issue).
3. Версионирование (semver-смысл):
   - **MAJOR** — удаление или переопределение существующего инварианта.
   - **MINOR** — добавление нового инварианта или enforcement-механизма.
   - **PATCH** — уточнение формулировки / «как проверить» / ссылок без смены сути.

**Молчаливый обход инварианта — нарушение.** Если решение объективно требует нарушить инвариант — сначала поправка устава, потом решение.

### Enforcement

- **Spec-гейт (Constitution Check):** Stage-2 audit прогоняет спеку/план через [constitution-check.md](constitution-check.md) — любой `MUST-FLAG` (или `NEEDS-INFO` на MUST) блокирует переход к разработке.
- **PR-гейт:** ревьюер (человек или агент) проходит ту же процедуру против диффа; нарушение без поправки = блок мержа.
- **QA-гейт:** `xt-protocol-qa` verify-gate — реальный JSON-RPC-прогон + adversarial + проверка секретов перед закрытием тикета (P-15).
