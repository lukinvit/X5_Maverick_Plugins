## 0. Почему планка enterprise

Плагин держит **секреты доступа** к трекеру организации и **мутирует рабочие данные** (тикеты, статусы, комментарии). Данные и секреты sacred — их нельзя терять, искажать, дублировать или утекать. Отсюда базовый принцип:

> **Заплатка на симптом — запрещена. Каждый баг лечится в корне, системно и долговечно.**

---

## 1. Intake — баг пришёл от владельца в чат

Запускается **немедленно**, одним непрерывным циклом, без gate'ов на одобрение между фазами:

1. **Сразу** завести GitHub issue с полной структурой: `## Контекст` (дата + дословная цитата) · `## Симптомы` · `## Возможные причины` · `## Желаемое поведение` (чек-лист приёмки) · `## Severity` (дефолт **P0/P1** для прямого фидбэка владельца) · `## Связанные`.
2. В том же цикле диспатчить fix-пак на investigate + fix — приоритет ВЫШЕ backlog.
3. **`xt-protocol-qa` verify-gate** после фикса (реальный stdio JSON-RPC-прогон, multi-tool, adversarial).
4. Закрывать **только** с evidence-транскриптом JSON-RPC.

**Железное правило:** каждый фидбэк владельца доводится в одном цикле, не откладывается на approval-gate.

---

## 2. ОБЯЗАТЕЛЬНО — `superpowers:systematic-debugging` на ЛЮБОЙ баг

При **любом** найденном дефекте, при **любой** критичности (P0 / P1 / P2 / P3) — прежде чем предлагать или писать фикс — прогнать через skill **`superpowers:systematic-debugging`**.

- Никаких «и так понятно» / «сразу в код». **Тривиальность симптома ≠ тривиальность причины.**
- Порядок: **воспроизвести → изолировать → найти root cause → доказать гипотезу (тест / JSON-RPC-прогон / лог) → только потом фикс.**
- Это касается и однострочных опечаток: систематическая отладка подтверждает, что однострочник и есть настоящая причина.
- **Фикс без доказанного root cause = не фикс. Не закрывать.**

Правило безусловное: не отменяется ни срочностью, ни «очевидностью», ни низкой критичностью.

---

## 3. Enterprise-решение, а не заплатка

- Лечи **причину**, а не симптом.
- Выбирай **архитектурно правильное и долговечное** решение, даже если оно дороже быстрого workaround.
- Workaround допустим **только** как явно помеченный временный (`# TEMP: <причина>, см. #<issue>`) со ссылкой на тикет системного фикса. **Молчаливая заплатка — нарушение** (P-13).
- Если правильное решение крупное — заведи issue/спеку на системный фикс и доведи через `/pipeline-lite`; не оставляй симптоматическую правку как финал.

---

## 4. Scope-the-class — бери весь класс в скоуп

Для каждого бага **обязательный вопрос:** «Может ли это повторяться в других частях плагина?»

Если да:

- Найди **ВСЕ** вхождения паттерна (grep / обзор всех tool-handlers), а не только репортнутое место.
- Почини **КАЖДУЮ** ось класса:
  - **tool** — повторяется ли дефект в других tools (тот же баг в `update_issue` и `transition_issue`)?
  - **поле schema** — другие поля с той же проблемой валидации/описания?
  - **роль XTracker** — проявляется ли иначе под другой ролью/правами инстанса (RBAC)?
  - **состояние иссью** — open / closed / без assignee / без статуса-перехода?
  - **поверхность вывода** — text / structured_content (card vs card_list) / compact-результат / ошибка?
- Поставь **guard против регрессии:** протокол-тест, проверка схемы, структурный barrier.
- Регистрируй как **defect-family** и веди до полного закрытия класса, а не одного симптома.

Один фикс по одной оси, когда осей пять, — это частичное покрытие.

## 5. Три сквозных стража каждого фикса: контракт-платформы / LLM-эргономика / безопасность-секретов

Любой фикс обязан **не ухудшать** систему по трём осям. Проверять **до** закрытия:

### Контракт платформы (P-4, P-5, P-6, P-7)
- `jarvis-plugin.json` валиден; zip-контракт цел.
- stdio JSON-RPC корректен; процесс не падает на новых/битых args.
- `slug` и `remote_name` инструментов не изменены (frozen identifiers).
- `inputSchema` строгая: `type:object`, `additionalProperties:false`, описания полей.

### LLM-эргономика (P-9, P-10, P-11)
- `SKILL.md` discovery не сломан (description в лимите, без литеральных вызовов tools).
- Нейминг `slug__verb_noun`; data-getters — защищённые verb'ы; action-tools — compact `(N)`-результат.
- `structured_content` всегда с text-fallback и не утекает в LLM-контекст; actions с полным tool-именем.
- Idempotent-skip соблюдён (дубль → `isError:false` + `⏭️ ПРОПУСК`).

### Безопасность-секретов (P-1, P-8)
- Секрет XTracker не появляется в `content`/`_meta`/логах/issue/памяти.
- RBAC XTracker не обойдён; 403 показан честно, не маскируется и не fabricated.

## 6. Никому не верь на слово — всё перепроверяй сам (evidence-first, zero-trust)

**Все врут** — репортёр бага, агент и его «готово / FIXED», зелёный прогон, и даже твои прошлые выводы и память. Это не про злой умысел — про ошибки, оптимизм, устаревший контекст. Доверие = дефект процесса (P-15).

- **Любое утверждение о результате** принимается ТОЛЬКО с воспроизводимым доказательством, которое **ты получил и осмотрел сам**: свежий repro, прогон протокол-теста с выводом, **реальный stdio JSON-RPC-раунд-трип** (запрос → ответ, осмотрен), grep по логу. Нет evidence — claim не существует.
- **Скепсис к агенту-разработчику особенно:** «FIXED» без свежей ревалидации — не доверять. Проверяй git landing (`git log`), не слова.
- **Скепсис к себе:** память — point-in-time, устаревает; верь живому коду/API, не воспоминанию. Эндпоинт/символ из памяти сверь с живым кодом перед использованием.
- **Изолированный PASS ≠ сквозной PASS:** «схема валидна» не доказывает, что tools/call под реальным инстансом отдаёт правильные данные; «работает на open-иссью» не доказывает closed.
- **Перепроверяй на другой оси, чем источник claim'а:** разработчик «проверил happy-path» → ты гоняешь adversarial (битые/недостающие args); «работает под админ-ключом» → проверь под ограниченной ролью.

## 7. Fix → test → verify → close (общий loop)

1. **Diagnose** — §2 (systematic-debugging), доказанный root cause.
2. **Decide** — trivial (один tool/handler, < 50 LoC) vs `/pipeline-lite` (новый tool / новая схема / cross-tool); ambiguous → `superpowers:brainstorming` до спеки. Всегда §3 (enterprise) + §4 (scope-the-class).
3. **Test** — протокол-харнесс (tools/list + tools/call happy + adversarial); `pytest -q`; `ruff check`; `mypy`. Для RBAC-чувствительных изменений — прогон под ограниченной ролью инстанса (при доступности инстанса).
4. **Verify** — `xt-protocol-qa` реальным JSON-RPC: tool вызывается без краша процесса; результат корректен; idempotent-skip соблюдён; секрет не утёк; structured_content имеет text-fallback. **«tools/list ок» без реального tools/call — не verify.**
5. **Close** — только с published SHA + протокол-тест + JSON-RPC evidence-транскрипт. Дубликат / defer проговаривается явно, не описывается как fixed.

---

## 8. Definition of Done (объединённый self-audit перед `gh issue close`)

Закрывать **только** если на все ответы «да»:

1. Root cause доказан через `superpowers:systematic-debugging` (§2)?
2. Решение системное/долговечное, не заплатка; workaround (если есть) помечен и тикетирован (§3)?
3. Проверено, повторяется ли класс; весь класс в скоупе (5 осей §4); поставлен guard от регрессии?
4. Три стража (контракт-платформы / LLM-эргономика / безопасность-секретов) — не ухудшены, проверены (§5)?
5. Claim «готово/FIXED» перепроверен СОБСТВЕННЫМ свежим evidence, желательно на другой оси (§6)?
6. Протокол-тесты + `pytest`/`ruff`/`mypy` — зелёные?
7. Реальный tools/call-прогон выполнен (не только валидация схемы)?
8. Секрет XTracker не утекает в любой вывод/лог (P-1)?
9. `xt-protocol-qa` verify-gate — PASS, с JSON-RPC-транскриптом?
10. Cross-surface consistency (text ↔ card ↔ card_list ↔ compact ↔ error) — без расхождений?

Если хоть один «нет» → **не закрывать.** Reopen / blocker-comment / лог в память.
