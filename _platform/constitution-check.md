# Constitution Check — gate-процедура

> Адаптировано из X5_BM mothership@c87e9ee6 (2026-06-13). Ядро портировано, слой локализован под MCP-плагины/XTracker.

**Статус:** canonical · операционный канон enforcement'а конституции (см. [constitution.md](constitution.md), Часть III).
**Версия:** 1.0.0 (2026-06-17).

**Что это.** Воспроизводимая процедура, которой Stage-2 audit, ревьюер PR или агент сверяют спеку / план / дифф с Частью I конституции (инварианты P-1…P-16). Это «авто-гейт» enforcement'а: переводит сверку из «если ревьюер вспомнил» в обязательный шаг с правилом блокировки.

## Принцип DRY

Процедура **не дублирует** формулировки инвариантов. Единственный источник истины — [constitution.md](constitution.md), Часть I. Здесь описано только КАК проверять и КАК блокировать. Путь устава — `docs/constitution.md` (одна константа; при переносе файла обнови ссылку здесь и в Stage-2).

## Вход

Любой из: `spec.md`, `plan.md`, дифф / PR, или текстовое описание изменения (например, спека нового tool'а).

## Процедура

1. Загрузи [constitution.md](constitution.md) — Часть I (P-1…P-16) и Часть III (precedence).
2. Для **каждого** инварианта P-1…P-16 оцени вход по его строке «_Как проверить_»:
   - **PASS** — соответствует, либо обоснованно неприменим;
   - **N/A** — инвариант не затрагивается изменением (укажи, почему);
   - **FLAG** — изменение нарушает или вероятно нарушит инвариант (укажи, где и чем);
   - **NEEDS-INFO** — по входу инвариант оценить нельзя.
3. Не угадывай. Если данных не хватает — `NEEDS-INFO` + требование уточнить спеку, а не молчаливый PASS (P-15).

## Выход (обязательный формат)

| Инвариант | Severity | Вердикт | Обоснование |
|---|---|---|---|
| P-1 Secrets Sovereignty | MUST | PASS/N/A/FLAG/NEEDS-INFO | … |
| … (все 16) … | | | |
| P-16 Bounded Results | SHOULD | … | … |

**Итог:** `MUST-FLAG: N · SHOULD-FLAG: N · NEEDS-INFO: N`.

> Severity берётся из самого инварианта: P-16 — единственный со SHOULD; остальные MUST.

## Правило блокировки

- **Любой `MUST-FLAG` = блокер этапа.** Спека не проходит Stage-2 (и PR не мержится), пока нарушение не устранено в спеке **ИЛИ** не проведена явная versioned-поправка устава (Часть III). Молчаливый обход запрещён (P-13 / P-15).
- **`NEEDS-INFO` на MUST-инвариант** = блокер до уточнения (непроверенное не пропускаем — P-15).
- **`SHOULD-FLAG`** (только P-16) — предупреждение: фиксируется в плане → «Known Risks», не блокирует.

## Self-тест (приёмка гейта)

Гейт обязан на этих эталонах давать ожидаемый результат, иначе он сломан:

- **`spec-bad`** — спека tool'а, которая: берёт XTracker-токен из переменной окружения `XTRACKER_TOKEN` (вместо `__jarvis.config`); объявляет `inputSchema` без `additionalProperties:false` и без `description` у полей; делает `delete_issue` без `confirm`-флага по умолчанию. Ожидаемо: **FLAG P-1** (секрет из env), **FLAG P-7** (нестрогая схема), **FLAG P-2** (деструктив по умолчанию); `MUST-FLAG ≥ 3`.
- **`spec-clean`** — спека read-tool'а `get_issue`: секрет берётся из `__jarvis.config`, `inputSchema` строгая (`additionalProperties:false`, описания полей, `required`), имя `get_issue` (защищённый verb, P-10), `text`-fallback к structured_content (P-11), `limit` не нужен (одиночный ресурс). Ожидаемо: все MUST = PASS/N/A; `MUST-FLAG: 0`.

Если `spec-bad` даёт <3 MUST-FLAG или `spec-clean` даёт >0 — гейт настроен неверно, чинить до использования.
